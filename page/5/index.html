枚举 EnumSet 详解

枚举类构造函数

  • 只能使用 private 修饰符
  • 即使不写，默认也是 private（不是包访问级别）

编译器生成的方法

  • values() — 返回所有枚举常量数组，每个枚举类独有
  • valueOf(String) — 简化的 valueOf 方法，内部调用 Enum.valueOf(Class, String)

EnumSet 实现原理

EnumSet 使用**位向量（bit vector）**实现，本质是一个 long 型数组。

存储结构

  • 每个枚举常量占 1 位（bit）
  • ≤64 个枚举 → 用单个 long

  • 64 个枚举 → 用 long[]（JumboEnumSet）

核心操作

添加 add:

把对应 bit 位置为1。

删除 remove:

把对应 bit 位清零。

判断 contains:

检查对应 bit 位是否为1。

原理图

性能特点

特点 说明
空间效率 每个枚举值只占 1 位
时间效率 添加/删除/查询都是 O(1) 位运算
顺序 按枚举声明顺序存储

remove 中 &=~ 与 ^= 的区别

元素是否在集合中 &= ~(1L<<n) (remove) ^= (1L<<n) (toggle)
在 (bit=1) 变成 0 ✓ 变成 0 ✓
不在 (bit=0) 保持 0 ✓ 变成 1 ✗

^= 是 toggle 操作，会把不存在的元素错误地添加进来。

相关类

  • RegularEnumSet — 枚举 ≤64 个
  • JumboEnumSet — 枚举 >64 个

问题

Java 8 之后允许接口出现 default 方法 和 static 方法，本质上是为了解决两个不同但都很重要的痛点。

下面用最直接的方式说明「为什么加」以及「加了之后主要解决什么问题」。

1. default 方法（默认方法）为什么允许存在？

最核心、最真实的动机只有一个：

让已有接口在不破坏海量已有实现类的情况下进化（向后兼容 + API 演进）

经典场景（也是 Java 8 引入 default 方法的真正导火索）

Java 8 要引入函数式编程 → Stream API → 需要在集合框架里加一堆新方法：

如果没有 default 方法，你加上面这些方法的结果就是：

  • 所有实现 Collection 的类（包括用户自己写的、第三方库写的）全部编译失败
  • 全世界 Java 代码瞬间崩掉几亿行

所以 default 方法的真正意义是：

  • 允许库作者在已有接口上安全地添加新行为
  • 已有实现类不用改动、不用重新编译 → 二进制兼容
  • 实现类可以选择「覆盖」这个默认实现，也可以直接用默认的

其他常见好处（次要，但也很实用）

  • 把“模版方法模式”从抽象类解放出来，能用接口实现（多继承友好）
  • 给函数式接口加便利方法（forEach、removeIf、replaceAll 等）
  • 减少工具类（如 Collections、Arrays）的出现，把工具方法写进接口里更内聚

一句话总结 default 方法的出现理由：

“我们想给 Collection、List、Map 等核心接口加 lambda/Stream 相关的新功能，但又不能让全世界代码都编译不过去，所以只好让接口自己带默认实现。”

2. static 方法为什么允许出现在接口里？

核心动机：把“属于这个接口的工具方法”写在接口旁边，而不是散落到工具类里

以前的痛点举例

这些静态工具方法逻辑上明明跟某个接口高度相关，却只能塞到 Collections、Arrays、Comparator.comparing 这种工具类里，显得分散、不内聚。

Java 8 之后可以这样写：

static 接口方法的优点总结

  • 高内聚：跟接口强相关的工厂方法、工具方法可以直接写在接口里
  • 命名空间更清晰：直接写 Comparator.naturalOrder() 而不是 Comparators.naturalOrder()
  • 减少无意义的工具类（Collections、Arrays 这种“万能工具袋”变少）
  • 不能被覆盖，天然安全（不像 default 方法可能被子类改写）

一句话总结 static 方法的出现理由：

“让真正属于这个接口的静态工厂方法、工具方法有个合理的家，而不是永远塞在某个 XXXs 工具类里。”

快速对比表（面试/复习用）

特性 default 方法 static 方法 出现的主要目的
有方法体 有 有 —
可被实现类覆盖 可以（override） 不可以 —
调用方式 实例.方法() 或 接口名.super.方法() 接口名.方法() —
主要解决的问题 接口进化时的二进制兼容 工具方法/工厂方法的归属清晰 默认 → 兼容老代码 static → 内聚
典型代表 Collection.stream() List.sort() Comparator.comparing() Optional.of() —

简单记忆口诀：

  • default = “默认给你一个实现，让接口能安全加功能，不让老代码爆炸”
  • static = “让接口自己带工具方法和工厂，别再到处 new XXXUtils 了”

各种访问权限

1, 类（普通类，抽象类）只能是包访问权限或者public的，不能是private或者protected的。内部类可以有上面四种
2, 抽象类的抽象方法，默认包访问权限, 也可以是public和protected的，不允许是private的
3, 接口可以是包访问权限或者public的
4, 接口的所有方法都是abstract的（除了default的），只能是public的
5, 接口的default方法必定是public的，default不是访问权限的一种，访问权限只有public，protected, package-private, private四种

Arch Linux 从 ext4 重装迁移到 Btrfs 方案

Arch Linux 从 ext4 重装迁移到 Btrfs 方案

目标：在保留 Windows 11 双系统的前提下，重装当前 Arch Linux，把根分区从 ext4 迁移到 Btrfs，并尽可能保留当前已安装软件、系统配置、用户配置、自定义环境变量等信息。

结论：基于当前机器的实际结构，最稳妥的方案是：保留 nvme0n1 上的整个 Windows 盘完全不动，只重装 nvme1n1 上的 Arch；复用当前 Arch 自己的 EFI 分区 /dev/nvme1n1p1，把根分区 /dev/nvme1n1p2 改成 Btrfs，继续保留单独的 /home 分区 /dev/nvme1n1p3。

1. 迁移目标

需要保留的内容：

  • 已安装软件列表
  • AUR 软件列表
  • /etc 中的重要系统配置
  • 用户目录中的配置与数据
  • shell / systemd 用户环境中的自定义环境变量
  • systemd 服务、timer、开发工具链配置
  • SSH / GPG / 字体 / 输入法 / 编辑器 / 终端等个人环境

需要实现的内容：

  • 仅把 Arch 根分区 /dev/nvme1n1p2 从 ext4 改成 Btrfs
  • 继续保留单独的 /home 分区 /dev/nvme1n1p3，不在本次重装中格式化
  • 继续复用当前 Arch 的 EFI 分区 /dev/nvme1n1p1，挂载点仍为 /efi
  • Windows 11 所在整块磁盘 nvme0n1 保持不动
  • 启动方式继续沿用当前的 GRUB + UEFI

2. 总体方案

分五步执行：

  1. 盘点当前系统状态
  2. 做完整备份
  3. 重装 Arch 到 Btrfs
  4. 恢复软件和配置
  5. 逐项验收

推荐采用保守方案：

  • 先恢复包
  • 再恢复配置
  • 最后恢复大数据目录

这样最容易排错，也最不容易把旧系统问题原样带回新系统。

3. 重装前导出系统状态

建议先准备一个备份目录，例如外置盘或其它分区挂载到 /mnt/backup。

3.1 导出软件列表

说明：

  • pkglist-explicit.txt：显式安装的软件
  • pkglist-foreign.txt：非官方仓库软件
  • pkglist-aur.txt：通常主要是 AUR 软件
  • pkglist-full.txt：完整留档

3.2 导出 systemd 服务信息

3.3 导出分区和启动信息

这台机器当前的实际分区结构如下：

  • nvme0n1：Windows 11 所在磁盘
    • nvme0n1p1：Windows EFI，100M，vfat
    • nvme0n1p2：MSR，16M
    • nvme0n1p3：Windows 系统分区，NTFS，199G
    • nvme0n1p4：Windows Recovery，NTFS，958M
    • nvme0n1p5：数据分区“新加卷”，NTFS，276.9G
  • nvme1n1：当前 Arch 所在磁盘
    • nvme1n1p1：Arch EFI，1G，vfat，当前挂载 /efi
    • nvme1n1p2：Arch 根分区，300G，ext4，当前挂载 /
    • nvme1n1p3：Arch /home 分区，1.6T，ext4，当前挂载 /home

当前 EFI 启动项：

  • Boot0003：GRUB，位于 nvme1n1p1 上的 \\EFI\\GRUB\\grubx64.efi
  • Boot0004：Windows Boot Manager，位于 nvme0n1p1

当前引导顺序是：

  • BootOrder: 0003,0004,...

这意味着当前机器已经是“Windows 一块盘 + Arch 一块盘”的双盘双系统结构，重装 Arch 时没有必要碰 Windows 那块盘。

5. 新系统的 Btrfs 布局建议

基于当前机器的实际情况，最推荐的做法不是把整个系统重新分很多 Linux 分区，而是：

  • 保留现有的 nvme1n1p1 作为 EFI
  • 把 nvme1n1p2 从 ext4 重建为 Btrfs，专门作为根系统盘
  • 保留现有的 nvme1n1p3 继续作为独立 /home 分区

也就是说，重装后的磁盘角色建议为：

  • /dev/nvme1n1p1 → /efi，vfat，不格式化
  • /dev/nvme1n1p2 → /，Btrfs，格式化
  • /dev/nvme1n1p3 → /home，ext4，不格式化

这样有几个现实好处：

  • 迁移改动面最小，只改根分区
  • 1.6T 的 /home 无需搬空后重建文件系统
  • 你现有大量用户数据、Docker bind mount、项目目录、缓存目录仍保持在单独 /home 分区上
  • 重装时最不容易误伤 Windows 或大体量数据

推荐在 nvme1n1p2 上创建这些 Btrfs 子卷：

  • @ → /
  • @var_log → /var/log
  • @pkg → /var/cache/pacman/pkg
  • @snapshots → /.snapshots

因为当前 /home 本来就是独立分区，所以这里不再在根 Btrfs 里创建 @home。

可选子卷：

  • @docker → /var/lib/docker
  • @libvirt → /var/lib/libvirt
  • @docker_root → /home/bkx/environment/docker_root

这样方便以后做快照管理，并把根目录、home、日志、包缓存拆开。
