字符串

字面量和新建对象

特性 “asdf” (字面量) new String(“asdf”)
内存位置 字符串常量池 (String Pool) 堆内存 (Heap)
对象数量 最多创建 1 个对象 至少创建 2 个对象（如果池中没有）
复用性 高（全局共享一份） 低（每次 new 都会产生新对象）
== 比较 指向池中同一个地址，为 true 指向堆中不同地址，为 false

内存布局拆解

当你执行这两行代码时，后台发生了完全不同的故事：

A. String s1 = "asdf";

  1. Java 检查字符串常量池。

  2. 如果 "asdf" 已经存在，直接返回引用。

  3. 如果不存在，在池中创建一个，然后返回引用。 这叫“驻留”（Interning），目的是为了省内存。

B. String s2 = new String("asdf");

  1. 编译器首先会在常量池中放入 "asdf"（如果不存在）。

  2. 在运行时，new 关键字强制在**堆（Heap）**中开辟一块全新的空间。

  3. 把常量池中 "asdf" 的内容拷贝一份到堆里。

  4. 返回堆内存的地址。

intern()方法

将字符串放入常量池中，比如两个字符串拼接之后，然后返回这个引用

底层存储结构改变

在 Java 9 之前，字符串内部是用 char[] 数组存储的。由于 Java 采用 UTF-16 编码，每个 char 强制占用 2 个字节（16位）。 但在实际开发中，绝大多数字符串其实只包含 Latin-1 字符（如英文、数字、简单符号），这些字符用 1 个字节就能存下。这意味着以前一半的内存空间都被浪费（填了 0）了。

为了解决这个问题，从 Java 9 开始引入了 “紧凑字符串”（Compact Strings）。

1. 底层结构的改变

Java 修改了 String 类的内部实现，将 char[] 换成了 byte[]，并增加了一个 coder 标志位。

  • 旧版 (JDK 8 及以前)： private final char value[]; —— 每个字符死板占用 2 bytes。

  • 新版 (JDK 9 至今)： private final byte[] value; —— 灵活存储。 private final byte coder; —— 记录编码格式（0 代表 Latin-1，1 代表 UTF-16）。

2. 它是如何工作的？

当你创建一个字符串时，Java 会自动判断：

  • 全英文/数字：coder 设为 0，每个字符占 1 byte。内存占用直接减半！

  • 含中文或其他特殊字符：coder 设为 1，退回到 UTF-16 模式，每个字符占 2 bytes（逻辑上和以前一样）。

这种切换对开发者是完全透明的，你不需要修改任何代码，升级 JDK 就能享受到内存红利。

3. 性能影响：变慢了吗？

很多人担心：每次操作字符串都要检查 coder 标志位，性能会不会下降？

结论是：基本没有，甚至更快。

  • 空间换时间：由于内存占用变小，GC（垃圾回收）的压力大幅减轻，整个系统的吞吐量反而提升了。

  • 向量化优化：现代 CPU 对 byte[] 的处理有非常强的指令集优化（SIMD），在字符串比较和拷贝时速度极快。

4. 一个直观的例子

假设你有一个包含 100 万个简单 ID 的列表："ID123456..."。

  • JDK 8：大约占用 40MB 内存。

  • JDK 9+：大约占用 20MB 内存。

在大型微服务应用中，字符串通常占堆内存的 25%-40%，这项优化能让整体内存占用下降 10%-15%，效果非常显著。

3.9.1 指数记数法

我记得指数记数法是有一个国际标准的，小数点多少位，指数多少位，怎么记录这些数字？

单精度和多精度（通常指双精度、四倍精度等）数字表示法的国际标准是 IEEE 754（全称 IEEE Standard for Floating-Point Arithmetic）。

该标准由电气电子工程师学会 (IEEE) 制定，并被国际标准化组织 (ISO) 和国际电工委员会 (IEC) 采纳为国际标准，编号为 ISO/IEC/IEEE 60559。

📋 标准核心内容

IEEE 754 规定了计算机中浮点数的存储格式、算术运算、舍入规则以及异常处理（如溢出、被零除、NaN 等）。

1. 常见精度格式

在 IEEE 754 标准中，最常用的两种格式是单精度和双精度：

格式名称 别名 总位数 符号位 (Sign) 指数位 (Exponent) 尾数位 (Mantissa) 十进制精度
单精度 binary32 32 bit 1 bit 8 bit 23 bit ~7 位有效数字
双精度 binary64 64 bit 1 bit 11 bit 52 bit ~16 位有效数字
四倍精度 binary128 128 bit 1 bit 15 bit 112 bit ~34 位有效数字

2. 数值的表示公式

一个浮点数的值通过以下公式计算：

$$Value = (-1)^S \times (1.f) \times 2^{E - Bias}$$

  • $S$ (Sign)：符号位。0 代表正数，1 代表负数。

  • $f$ (Fraction/Mantissa)：尾数的小数部分。隐含的高位为 1（规格化数）。

  • $E$ (Exponent)：阶码（指数）。

  • $Bias$ (偏移量)：为了表示负指数，标准规定了一个偏移值。单精度为 127，双精度为 1023。

🔍 为什么这个标准很重要？

  1. 跨平台一致性：无论你在 Intel CPU、ARM 架构还是英伟达 GPU 上运行，同样的浮点运算代码应该得到相同的结果。

  2. 特殊的数值处理：标准定义了如何表示特殊状态：

    • NaN (Not a Number)：用于非法运算（如 $\sqrt{-1}$）。

    • Infinity ($\infty$)：用于表示超出范围的极值。

    • $\pm 0$：区分正零和负零。

  3. 舍入模式：规定了四种（或更多）舍入规则，确保精度损失是可控且可预测的。

Java 泛型 (Generics) 核心笔记

Java 泛型 (Generics) 核心笔记

1. 泛型的本质：参数化类型 (Parameterized Types)

泛型是将类型本身作为参数传递给类、接口或方法。其核心目标是：在编译期进行类型检查，减少运行时的类型转换异常 (ClassCastException)。

优势

  1. 类型安全：编译器会拦截错误的类型插入。
  2. 消除强制转换：代码更简洁，可读性更强。
  3. 代码重用：编写一套逻辑，处理多种数据类型（如 ArrayList<T>）。

2. 核心难点：类型擦除 (Type Erasure)

这是 Java 泛型最独特、也最受争议的特性。

原理

Java 的泛型只存在于编译期。为了兼容旧版本的 JVM，编译器在生成字节码时，会将所有的泛型参数“擦除”：

  • 如果没有边界，替换为 Object。
  • 如果有边界（如 <T extends Number>），替换为边界类型（Number）。

擦除带来的限制

  1. 运行时无法获取泛型类型：无法执行 if (obj instanceof T)。
  2. 不能创建泛型数组：无法执行 new T[10]。
  3. 基本类型不能作为泛型参数：必须使用包装类（如 List<Integer> 而非 List<int>）。

3. 通配符与 PECS 原则

这是泛型进阶中最实用的规则，决定了泛型集合的读写权限。

PECS 原则 (Producer Extends, Consumer Super)

  1. Producer Extends (<? extends T>)：

    • 场景：如果你只需要从集合中读取数据（集合是生产者）。
    • 权限：可以安全读取为 T 类型，但不能写入任何对象（除了 null）。
    • 意义：确定了上限。

  2. Consumer Super (<? super T>)：

    • 场景：如果你需要往集合中写入数据（集合是消费者）。
    • 权限：可以安全写入 T 及其子类，但读取出来只能是 Object 类型。
    • 意义：确定了下限。

代码对比

4. 泛型方法与泛型类

  • 泛型类：public class Box<T> { ... } (T 在实例化类时确定)
  • 泛型方法：public <E> void process(E element) { ... } (E 在调用方法时根据参数推断)

switch选择因子

在 Java 的演进过程中，switch 能够处理的数据类型经历了几次重大的扩张。到了 Java 25，由于引入了模式匹配（Pattern Matching），几乎任何类型都可以作为 switch 的选择因子。

以下是按 Java 版本演进划分的类型支持：

1. 基础类型（JDK 1.0 - 5.0）

这是最传统的 switch 支持，要求选择因子必须是能够自动转换为 int 的整型或特定类型：

  • byte, short, char, int

  • 包装类：Byte, Short, Character, Integer（由于自动拆箱机制）

  • enum (枚举类型)：从 JDK 5 开始支持。

2. 字符串类型（JDK 7）

  • String：从 JDK 7 开始，你可以直接对字符串进行分支判断。

    • 注意：底层是通过字符串的 hashCode() 和 equals() 实现的。

3. 模式匹配类型（JDK 17 - 21/25）

这是 Java 现代化的核心。现在的 switch 不再仅仅是“相等判断”，而是“类型与结构的匹配”。

  • 任何对象引用：你可以传入 Object 类型的变量。

  • Record 类型：支持解构（Destructuring），可以直接匹配 Record 内部的字段。

  • null：在以前的版本中，传入 null 会抛出 NullPointerException。但在现代 Java 中，你可以显式写出 case null。

圆柱倒角问题

blender中新建了一个6边的圆柱体，对顶部的边进行倒角的时候，圆柱体的侧面没有参与倒角，只在圆柱体顶面有倒角，原因是什么？

1. 核心原因：顶点没有合并（重叠顶点）

这是最常见的情况。在创建圆柱体或进行某些操作（如错误的挤出）时，圆柱体的“顶面”和“侧面”虽然看起来在一起，但它们的点和点之间其实没有焊接上。

  • 表现：倒角时，Blender 认为你只选中了一个独立的“圆盘”边缘，它只会向圆盘中心收缩，而不会拉动侧面的网格。

解决方法：

  1. 进入编辑模式 (Tab)。

  2. 全选所有点 (A)。

  3. 按下 M 键，选择 按距离合并 (Merge by Distance)。

  4. 观察屏幕下方是否显示“删除了 X 个顶点”。如果删除了点，再尝试倒角就正常了。

2. 物体缩放比例未应用 (Scale)

如果你在“物体模式”下非等比例缩放过圆柱体（比如拉长了它），Blender 的倒角计算会发生畸变。

  • 表现：倒角可能变得非常扁，或者在某个轴向上完全没反应，看起来就像侧面没参与一样。

解决方法：

  1. 回到物体模式 (Tab)。

  2. 按下 Ctrl + A，选择 缩放 (Scale)。

  3. 你会发现右侧侧栏（快捷键 N）中的“缩放”数值全部变成了 1.0。

  4. 再次进入编辑模式进行倒角。

3. 开启了“钳制重叠” (Clamp Overlap)

如果你的圆柱体非常小，或者倒角量太大，倒角边缘会碰到其他的边。

  • 表现：倒角滑块拉不动，或者拉到一半就卡住了，导致你觉得侧面没变化。

解决方法：

  • 在执行倒角命令时，观察屏幕左下角的 倒角 (Bevel) 属性面板。

  • 检查是否勾选了 钳制重叠 (Clamp Overlap)。

  • 尝试关闭它，看看边线是否能继续移动（但注意关闭后可能会导致面与面穿插）。

4. 误用了“顶点倒角”

检查一下你按下的是不是 Ctrl + B。

  • Ctrl + B 是边倒角（正常切开转角）。

  • Ctrl + Shift + B 是顶点倒角。如果你误用了后者，它只会把每一个角变成一个小平面，而不会影响侧面的边。

总结排查步骤

如果你现在正对着那个圆柱体发愁，请按顺序执行以下“三部曲”，90% 的概率能解决：

  1. 物体模式下：Ctrl + A -> 应用缩放。

  2. 编辑模式下：A全选 -> M -> 按距离合并。

  3. 确认选择：确保你选中的是圆柱体顶部那圈边（边选择模式），而不是整个顶面。

**认识界面

什么是游标和原点

游标

模型中间圆圈叫做游标，两个作用
1。游标在哪里，你新建的物体就会出现在哪里，物体也可以移动到游标
2。游标可以作为物体旋转的轴心点使用

操作游标
点击左侧游标工具，然后鼠标左键点击到哪里游标就到哪里
快捷键：
Shift + 鼠标右键 点击到哪里游标就到哪里
Shift + S 选中物体之后，出现一个操作轮盘，选择到游标，物体就会移动到游标所在的位置
Shift + c 游标回到坐标系原点


默认物体旋转是按照每个物体的原点旋转，可以在变换轴心点中选择游标，之后会按照游标进行旋转

原点
鼠标点击物体选中之后，物体会出现一个黄色的点，就是物体的原点
物体缩放是从原点开始的，可以通过勾选下面的原点，移动物体的原点（移动快捷键G）

可以将原点移动到坐标系中心，之后取消物体移动（Alt + G），可以发现物体的原点回到了世界中心，物体本身和原点还保持着之前的相对关系，但是并没有在世界中心

也就是说程序只认识物体的原点，所有的操作都是按照原点操作

将原点移动到物体默认位置： 右键物体 -> 设置原点->原点到几何中心

在G移动模式下，按住ctrl移动，可以吸附网格进行移动

1, 边界框中心： 选中多个物体是，物体被最小的框包围时，框的中心点
2, 3D游标：坐标系上的圆圈
3, 各自的原点： 每个物体有自己的原点, 认为设置的橙色的点，
4, 质心点： 物体的重心，如果有多个物体是多个物体的重心（这里貌似blender翻译错误，英文是median point, 像是中心点，但不是质心点，实测不是中心，将原点移出物体之外，选择按照质心点旋转，仍然是按照原点旋转）
5, 活动元素： 选中多个元素时，最后选中的那个物体

坐标

1, 全局坐标，所有物体共有的坐标，界面上显示的x, y和z轴
2, 局部坐标，每个物体自己的坐标，跟随物体

快捷键
g x/y/z 局部坐标x/y/z
g x/y/z x/yz 全局坐标x/y/z

四种预览方式

1, 渲染预览，显示最终渲染结果预览，但是相对占资源，慢
2, 材质预览，可以预览材质的颜色和简单纹理，但是不计算光线追踪细节，比渲染预览快
3, 实体模式预览，不现实颜色和材质，建模时候使用，快，不占太多资源
4, 线框模式预览，可以显示内部和背面的走线，建模时候常用
