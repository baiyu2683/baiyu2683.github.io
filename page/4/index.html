Git 提交信息与拆分

Git 提交信息与拆分

这篇笔记解决什么问题

这篇笔记主要帮助解决两个问题：

  • 这次改动应该保留为一个 commit，还是拆成多个 commit
  • 这条 commit message 应该怎么写，才能让别人看得懂、以后也方便回看

目标不是追求形式上的完美，而是让提交更容易 review、更容易回滚，也更容易理解。

什么时候一个 commit 就够

通常满足下面这些条件时，一个 commit 就够：

  • 这次改动只服务于一个明确目标
  • 改动涉及的文件虽然不止一个，但仍然在讲同一件事
  • 你能用一句话说清这次提交为什么存在
  • 以后如果要回滚这次改动，不会顺便回滚掉无关内容

简单判断：

如果别人只看这一个 commit，就能理解你做了什么、为什么这么做，而且单独回滚也不会伤到别的内容，那通常一个 commit 就够。

什么时候应该拆成多个 commit

遇到下面这些情况时，通常应该拆：

  • 重构和功能修改混在一起
  • 机械性改动和逻辑改动混在一起
  • 两个不相干的问题被顺手一起改了
  • 某一部分改动本来就可以独立 review
  • 你自己已经开始说“顺手把这个也改了”

常见例子：

  • 一次提交里既改命名、又改逻辑、又补文档
  • 为了修一个 bug，顺手把另一个模块的格式也整理了
  • 同时包含“纯重构”和“真实行为变化”

拆 commit 的核心不是追求越多越好，而是看：

  • 能不能独立 review
  • 能不能独立回滚
  • message 能不能各自说清楚

怎么写 commit message

实操上，先做到两件事就够：

  1. 开头类型尽量明确
  2. 一句话能看出这次提交为什么存在

常见起手：

  • feat: 新功能或新能力
  • fix: 修 bug 或纠正错误行为
  • refactor: 重构，但不改变对外行为
  • docs: 文档说明调整

好的 message 应该让以后回看的人快速明白：

  • 这是在加东西、修东西，还是整理东西
  • 这次改动存在的原因是什么

例如：

  • feat: add manual launcher for dc-spider-tool
  • fix: correct worktree cleanup order after PR merge
  • refactor: simplify proxy startup entrypoint
  • docs: clarify when to split commits

尽量不要写这种过于空泛的标题：

  • update
  • fix bug
  • 修改一下
  • 调整代码

这些标题的问题不是“不规范”，而是几乎不能帮助别人理解这次提交为什么存在。

常见反例

反例 1：一个 commit 里混太多事

比如：

  • 改了代码逻辑
  • 顺手统一了格式
  • 又补了一点文档

这种提交的问题是：

  • review 时重点不清楚
  • 回滚时容易误伤
  • message 很难写准确

反例 2：message 太空

比如：

  • update
  • fix
  • 改一下

这种 message 的问题是，几天后连你自己都不一定知道当时到底改了什么、为什么改。

反例 3：拆得过碎

不是 commit 越多越好。

如果拆到每个 commit 都只剩下零散、不完整、无法独立理解的改动，review 反而更痛苦。

好的拆分应该让每个 commit 都还是一个完整、能讲明白的小单位。

提交前快速检查

提交前可以快速问自己这几个问题：

  • 这个 commit 是不是只在表达一件事
  • 别人只看这一个 commit，能不能大致理解意图
  • 如果现在回滚它，会不会顺便回滚掉不该回滚的内容
  • 这条 message 有没有说明为什么改，而不只是“改了什么”

如果这四个问题里有两个以上答不上来，通常就该停一下，重新整理 commit。

  • 这是新增能力吗？用 feat
  • 这是修错误吗？用 fix
  • 这是整理代码但不改行为吗？用 refactor
  • 这是改说明文档吗？用 docs

冲突处理与常见问题

冲突处理与常见问题

适用场景

这篇笔记解决的问题是：当 git pull、merge、rebase、stash、worktree 清理这些环节出问题时，应该先怎么判断、再怎么处理。

本地有未提交内容时，能不能直接 pull

默认不建议。

因为这时通常会发生两种情况：

  1. Git 直接拒绝 pull
  2. pull 进入 merge/rebase 后出现冲突

更稳妥的做法是先处理本地改动：

pull 被拒绝时怎么处理

优先按这个顺序判断：

  1. 你是不是在一个不该直接开发的分支上改了东西
  2. 这些本地改动是不是已经值得提交
  3. 如果只是临时内容，是否适合先 stash

大多数情况下，优先考虑：

  • 先 commit
  • 次选 stash
  • 再 pull

merge / rebase 冲突时怎么想

先不要急着做复杂操作，先确认：

  • 这是你自己的功能分支，还是 master
  • 你现在是在 merge 过程中，还是 rebase 过程中
  • 冲突文件里哪些改动是本地的，哪些来自远端

处理思路：

  • 如果是日常同步功能分支，通常优先把冲突解开，然后继续当前流程
  • 如果是 master 上出现混乱，先确认是不是因为你不小心在 master 上开发了

stash 什么时候用

适合用 stash：

  • 你只是临时切一下上下文
  • 当前改动还不值得形成正式提交

不建议滥用 stash：

  • 长时间堆很多 stash 不处理
  • 把本来应该提交的小步进度一直藏在 stash 里

worktree 删除失败怎么办

常见原因：

  • worktree 里还有未提交内容
  • 该 worktree 正被某个 shell 占用
  • 分支状态还没整理干净

先检查：

分支删不掉通常是什么原因

常见原因：

  • 该分支还有未合并提交
  • 当前正停留在这个分支上
  • 这个分支仍然被某个 worktree 使用

先检查：

常见问题速查

为什么 push 的不是 worktree？

因为远端识别的是 Git 分支，不是你本地目录。

为什么 master 要保持干净？

因为这样同步远端最简单，冲突最少。

什么时候必须新开分支？

只要是一个独立任务，就建议新开分支。

一个 PR 多大合适？

以“别人能看懂并能安全 review”为准，不要把多个无关需求塞进同一个 PR。

PR、评审与合并

PR、评审与合并

适用场景

这篇笔记解决的问题是：代码已经在功能分支里，接下来应该怎么发 PR、怎么 review、怎么合并。

什么时候发 PR

通常在以下情况发 PR：

  • 当前任务已经形成一段相对完整的改动
  • 代码已经可以让别人 review
  • 你准备进入合并流程

PR 里至少写什么

建议最少写清楚：

  • 这次改了什么
  • 为什么这样改
  • 还有没有已知限制
  • review 时最值得重点看的地方

小团队下的最低 review 要求

适合小团队的最低要求：

  • 改动较小且风险低时，可以快速 review
  • 改动较大、影响共享逻辑、或可能影响稳定性时，最好等另一个人看过
  • 如果只是你个人维护的内容，可以按自己的节奏合并，但仍建议保留 PR 记录

什么时候可以自己合并

通常可以自己合并：

  • 只有你自己在维护该模块
  • 改动很小，且影响范围清楚
  • 团队已经约定这类改动无需等待额外 review

通常不建议自己直接合并：

  • 改动涉及公共基础能力
  • 改动较大
  • 其他人正在并行依赖这部分代码

默认合并建议

小团队默认建议优先使用 squash merge。

原因：

  • PR 内可以保留开发过程中的多个小提交
  • 合并到 master 后历史更整洁
  • 回看主分支历史时更容易理解

如果团队明确希望保留完整分支提交历史，再考虑普通 merge。

常见误区

误区 1：PR 合并后，还要不要本地再 merge 一遍？

通常不需要。

误区 2：是不是把 worktree 推到远端？

不是。

推到远端的是分支，worktree 只是本地目录。

对小团队来说，不需要过度流程化，但至少要保证别人能看懂改动目的。

团队协作总览

团队协作总览

这套手册适合谁

这套手册适合我自己和少量协作者一起开发时使用。

目标是：

  • 日常开发步骤尽量固定
  • 减少因为分支、worktree、PR 使用不一致带来的混乱
  • 遇到常见 Git 问题时能快速查到处理办法

这不是大团队流程文档，也不是公司级规范，而是一套能直接执行的小团队协作手册。

我们的基本约定

  • master 保持干净，只负责同步远端
  • 一个任务一个分支
  • 一个任务尽量对应一个 worktree
  • 日常开发在功能分支中完成，不直接在 master 上改代码
  • 代码完成后先推分支，再发 PR
  • PR 合并通常在远端完成，本地只负责同步结果

分支、worktree、PR 的关系

  • 分支：表示一段独立开发工作
  • worktree：本地承载某个分支的一份独立目录
  • PR：把某个分支的改动提交给团队评审和合并

一句话理解：

worktree 是本地工作目录，真正推到远端的是分支，协作入口是 PR。

标准协作主线

  1. 先在主 worktree 中同步本地 master
  2. 为新任务创建分支和 worktree
  3. 在 worktree 中开发和提交
  4. 推送分支到远端
  5. 创建 PR
  6. 完成 review 与合并
  7. 回到主 worktree 同步 master
  8. 删除已完成的分支和 worktree

从哪里开始看

  • 想开始一个新任务：看 [[日常开发流程]]
  • 想发 PR 或决定怎么合并：看 [[PR、评审与合并]]
  • 遇到 pull、冲突、stash、清理失败等问题：看 [[冲突处理与常见问题]]

阅读顺序建议

  1. 先读这篇总览
  2. 日常开发时主要查 [[日常开发流程]]
  3. 提交协作时查 [[PR、评审与合并]]
  4. 出现异常时查 [[冲突处理与常见问题]]

java编程思想

内部类

匿名内部类使用外部变量，为什么必须是final的或者是“事实上的final”

匿名内部类在编译后会变成一个独立的类（比如 Outer$1.class），它和外部方法是两个不同的栈帧（不同的方法执行上下文）。

  • 匿名内部类对象通常会比外部方法活得更久（比如作为事件监听器、Runnable 提交到线程池、被返回出去等）。
  • Java 不允许匿名内部类直接修改外部方法的栈上局部变量（那样会导致数据不一致、线程安全隐患、生命周期问题）。
  • 为了实现“捕获”（capture），Java 的做法是：把用到的局部变量的值拷贝一份 到匿名内部类的实例字段中（编译器自动生成）。

如果允许修改外部变量，就会出现“拷贝的值”和“原变量”不一致的情况（经典的多线程可见性问题）。

所以 Java 强制：被捕获的局部变量在整个方法执行期间值不能改变，也就是必须是 final 或 effectively final。

基本数据类型

Java 中的基本数据类型（Primitive Types）共有 8 种。它们是由语言预定义的，直接存储在栈（Stack）中，因此在内存操作上非常高效。

我们可以根据它们存储的数据性质，将其分为 4 大类：

1. 整数类型 (Integer Types)

用于存储不带小数点的整数。Java 的整数类型都是有符号的（即支持正负数）。

类型 大小 (位/比特) 字节 (Bytes) 范围 默认值
byte 8-bit 1 -128 到 127 0
short 16-bit 2 -32,768 到 32,767 0
int 32-bit 4 约 ±21 亿 0
long 64-bit 8 非常大 (需以 L 或 l 结尾) 0L

2. 浮点类型 (Floating-Point Types)

用于存储带小数点的数值，遵循 IEEE 754 标准。

类型 大小 (位) 字节 精度 默认值
float 32-bit 4 单精度 (需以 f 或 F 结尾) 0.0f
double 64-bit 8 双精度 (Java 默认的小数类型) 0.0d

3. 字符类型 (Character Type)

用于存储单个字符。

  • char: 占用 16 位 (2 字节)。

  • 它使用 Unicode (UTF-16) 编码。

  • 范围：'\u0000' (即 0) 到 '\uffff' (即 65,535)。

  • 默认值：'\u0000'。

4. 布尔类型 (Boolean Type)

用于逻辑判断。

  • boolean: 只有两个可能的值：true 和 false。

  • 大小：JVM 规范并没有明确规定其占用的大小（通常在内存中占用 1 字节或 4 字节，取决于 JVM 的实现）。

  • 默认值：false。
