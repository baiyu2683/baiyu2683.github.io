PR、评审与合并

PR、评审与合并

适用场景

这篇笔记解决的问题是：代码已经在功能分支里，接下来应该怎么发 PR、怎么 review、怎么合并。

什么时候发 PR

通常在以下情况发 PR：

  • 当前任务已经形成一段相对完整的改动
  • 代码已经可以让别人 review
  • 你准备进入合并流程

PR 里至少写什么

建议最少写清楚：

  • 这次改了什么
  • 为什么这样改
  • 还有没有已知限制
  • review 时最值得重点看的地方

小团队下的最低 review 要求

适合小团队的最低要求：

  • 改动较小且风险低时，可以快速 review
  • 改动较大、影响共享逻辑、或可能影响稳定性时，最好等另一个人看过
  • 如果只是你个人维护的内容，可以按自己的节奏合并，但仍建议保留 PR 记录

什么时候可以自己合并

通常可以自己合并：

  • 只有你自己在维护该模块
  • 改动很小，且影响范围清楚
  • 团队已经约定这类改动无需等待额外 review

通常不建议自己直接合并：

  • 改动涉及公共基础能力
  • 改动较大
  • 其他人正在并行依赖这部分代码

默认合并建议

小团队默认建议优先使用 squash merge。

原因：

  • PR 内可以保留开发过程中的多个小提交
  • 合并到 master 后历史更整洁
  • 回看主分支历史时更容易理解

如果团队明确希望保留完整分支提交历史，再考虑普通 merge。

常见误区

误区 1：PR 合并后，还要不要本地再 merge 一遍？

通常不需要。

一般是远端 PR 合并完成后，你本地更新 master 即可。

误区 2：是不是把 worktree 推到远端？

不是。

推到远端的是分支，worktree 只是本地目录。

误区 3：是不是每个 PR 都必须很正式？

对小团队来说，不需要过度流程化，但至少要保证别人能看懂改动目的。
