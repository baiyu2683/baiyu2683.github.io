日常开发流程

适用场景

这篇笔记解决的问题是：从接到一个新任务开始，到把代码推到远端分支为止，日常应该怎么操作。

标准流程

1. 先在主 worktree 里同步主分支

这里的命令应该在主仓库所在的 master worktree 中执行，不要在当前功能 worktree 里直接切回 master。

要求：

  • master 尽量保持干净
  • 不要在 master 上直接开发

2. 为新任务创建分支和 worktree

含义：

  • 基于 master 创建 feature/foo
  • 在 ../wt-foo 目录中开始新任务

3. 在 worktree 里开发

常用节奏：

建议：

  • 小步提交
  • 一个提交只表达一件事
  • 不要等到改了一大堆再一次性提交

4. 首次推送远端分支

之后继续开发时，正常 git push 即可。

一个任务对应一个分支

推荐：

  • 一个任务一个分支
  • 一个任务一个 worktree

不推荐：

  • 多个需求混在同一个分支里
  • 在同一个 worktree 中长期混做多个任务

什么时候应该 push

推荐在这些时机 push：

  • 一个功能点已经可以保存
  • 想让别人提前看到进展
  • 准备发 PR
  • 需要把当前进度同步到远端备用

不要这样做

  • 不要在 master 上直接写业务代码
  • 不要在有大量未整理改动时才第一次提交
  • 不要把多个无关需求塞进同一个分支
  • 不要把 worktree 当成远端协作对象，真正协作的是分支
