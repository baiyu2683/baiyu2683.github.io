字符串

字面量和新建对象

特性 “asdf” (字面量) new String(“asdf”)
内存位置 字符串常量池 (String Pool) 堆内存 (Heap)
对象数量 最多创建 1 个对象 至少创建 2 个对象（如果池中没有）
复用性 高（全局共享一份） 低（每次 new 都会产生新对象）
== 比较 指向池中同一个地址，为 true 指向堆中不同地址，为 false

内存布局拆解

当你执行这两行代码时，后台发生了完全不同的故事：

A. String s1 = "asdf";

  1. Java 检查字符串常量池。

  2. 如果 "asdf" 已经存在，直接返回引用。

  3. 如果不存在，在池中创建一个，然后返回引用。 这叫“驻留”（Interning），目的是为了省内存。

B. String s2 = new String("asdf");

  1. 编译器首先会在常量池中放入 "asdf"（如果不存在）。

  2. 在运行时，new 关键字强制在**堆（Heap）**中开辟一块全新的空间。

  3. 把常量池中 "asdf" 的内容拷贝一份到堆里。

  4. 返回堆内存的地址。

intern()方法

将字符串放入常量池中，比如两个字符串拼接之后，然后返回这个引用

底层存储结构改变

在 Java 9 之前，字符串内部是用 char[] 数组存储的。由于 Java 采用 UTF-16 编码，每个 char 强制占用 2 个字节（16位）。 但在实际开发中，绝大多数字符串其实只包含 Latin-1 字符（如英文、数字、简单符号），这些字符用 1 个字节就能存下。这意味着以前一半的内存空间都被浪费（填了 0）了。

为了解决这个问题，从 Java 9 开始引入了 “紧凑字符串”（Compact Strings）。

1. 底层结构的改变

Java 修改了 String 类的内部实现，将 char[] 换成了 byte[]，并增加了一个 coder 标志位。

  • 旧版 (JDK 8 及以前)： private final char value[]; —— 每个字符死板占用 2 bytes。

  • 新版 (JDK 9 至今)： private final byte[] value; —— 灵活存储。 private final byte coder; —— 记录编码格式（0 代表 Latin-1，1 代表 UTF-16）。

2. 它是如何工作的？

当你创建一个字符串时，Java 会自动判断：

  • 全英文/数字：coder 设为 0，每个字符占 1 byte。内存占用直接减半！

  • 含中文或其他特殊字符：coder 设为 1，退回到 UTF-16 模式，每个字符占 2 bytes（逻辑上和以前一样）。

这种切换对开发者是完全透明的，你不需要修改任何代码，升级 JDK 就能享受到内存红利。

3. 性能影响：变慢了吗？

很多人担心：每次操作字符串都要检查 coder 标志位，性能会不会下降？

结论是：基本没有，甚至更快。

  • 空间换时间：由于内存占用变小，GC（垃圾回收）的压力大幅减轻，整个系统的吞吐量反而提升了。

  • 向量化优化：现代 CPU 对 byte[] 的处理有非常强的指令集优化（SIMD），在字符串比较和拷贝时速度极快。

4. 一个直观的例子

假设你有一个包含 100 万个简单 ID 的列表："ID123456..."。

  • JDK 8：大约占用 40MB 内存。

  • JDK 9+：大约占用 20MB 内存。

在大型微服务应用中，字符串通常占堆内存的 25%-40%，这项优化能让整体内存占用下降 10%-15%，效果非常显著。
