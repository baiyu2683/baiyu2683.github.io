GitHub Fork 标准工作流程

适用场景

这篇笔记解决的问题是：在 GitHub 上 fork 了别人的仓库之后，本地应该怎么配置远端、怎么同步源仓库代码、怎么开发功能分支，以及怎么避免误推到源仓库。

适合这种关系：

  • 源仓库：别人或组织维护的原始仓库
  • fork 仓库：自己账号下复制出来的仓库
  • 本地仓库：自己电脑上的工作目录

基本约定

推荐固定使用这两个远端名字：

  • origin：自己的 fork 仓库，可以 push
  • upstream：源仓库，只用来 fetch，不直接 push

一句话记忆：

从 upstream 拉代码，往 origin 推代码，最后通过 PR 请求合并。

为什么使用 –ff-only

--ff-only 的好处是：只有本地主分支能直接快进时才合并。

如果你不小心在本地 main 上提交了自己的开发内容，--ff-only 会拒绝合并，避免把 fork 的主分支历史弄乱。

误区 4：把本地 main 设置成跟踪 upstream/main

不建议在 fork 工作流里这样做：

这样会让后续 git push 的目标更容易变得不清楚。更稳妥的方式是：

  • 本地 main 跟踪 origin/main
  • 同步源仓库时显式写 git fetch upstream
  • 合并时显式写 git merge --ff-only upstream/main
