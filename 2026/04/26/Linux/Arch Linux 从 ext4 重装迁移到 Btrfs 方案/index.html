Arch Linux 从 ext4 重装迁移到 Btrfs 方案

目标：在保留 Windows 11 双系统的前提下，重装当前 Arch Linux，把根分区从 ext4 迁移到 Btrfs，并尽可能保留当前已安装软件、系统配置、用户配置、自定义环境变量等信息。

结论：基于当前机器的实际结构，最稳妥的方案是：保留 nvme0n1 上的整个 Windows 盘完全不动，只重装 nvme1n1 上的 Arch；复用当前 Arch 自己的 EFI 分区 /dev/nvme1n1p1，把根分区 /dev/nvme1n1p2 改成 Btrfs，继续保留单独的 /home 分区 /dev/nvme1n1p3。

1. 迁移目标

需要保留的内容：

  • 已安装软件列表
  • AUR 软件列表
  • /etc 中的重要系统配置
  • 用户目录中的配置与数据
  • shell / systemd 用户环境中的自定义环境变量
  • systemd 服务、timer、开发工具链配置
  • SSH / GPG / 字体 / 输入法 / 编辑器 / 终端等个人环境

需要实现的内容：

  • 仅把 Arch 根分区 /dev/nvme1n1p2 从 ext4 改成 Btrfs
  • 继续保留单独的 /home 分区 /dev/nvme1n1p3，不在本次重装中格式化
  • 继续复用当前 Arch 的 EFI 分区 /dev/nvme1n1p1，挂载点仍为 /efi
  • Windows 11 所在整块磁盘 nvme0n1 保持不动
  • 启动方式继续沿用当前的 GRUB + UEFI

2. 总体方案

分五步执行：

  1. 盘点当前系统状态
  2. 做完整备份
  3. 重装 Arch 到 Btrfs
  4. 恢复软件和配置
  5. 逐项验收

推荐采用保守方案：

  • 先恢复包
  • 再恢复配置
  • 最后恢复大数据目录

这样最容易排错，也最不容易把旧系统问题原样带回新系统。

5. 新系统的 Btrfs 布局建议

基于当前机器的实际情况，最推荐的做法不是把整个系统重新分很多 Linux 分区，而是：

  • 保留现有的 nvme1n1p1 作为 EFI
  • 把 nvme1n1p2 从 ext4 重建为 Btrfs，专门作为根系统盘
  • 保留现有的 nvme1n1p3 继续作为独立 /home 分区

也就是说，重装后的磁盘角色建议为：

  • /dev/nvme1n1p1 → /efi，vfat，不格式化
  • /dev/nvme1n1p2 → /，Btrfs，格式化
  • /dev/nvme1n1p3 → /home，ext4，不格式化

这样有几个现实好处：

  • 迁移改动面最小，只改根分区
  • 1.6T 的 /home 无需搬空后重建文件系统
  • 你现有大量用户数据、Docker bind mount、项目目录、缓存目录仍保持在单独 /home 分区上
  • 重装时最不容易误伤 Windows 或大体量数据

推荐在 nvme1n1p2 上创建这些 Btrfs 子卷：

  • @ → /
  • @var_log → /var/log
  • @pkg → /var/cache/pacman/pkg
  • @snapshots → /.snapshots

因为当前 /home 本来就是独立分区，所以这里不再在根 Btrfs 里创建 @home。

可选子卷：

  • @docker → /var/lib/docker
  • @libvirt → /var/lib/libvirt
  • @docker_root → /home/bkx/environment/docker_root

这样方便以后做快照管理，并把根目录、home、日志、包缓存拆开。

6. 重装时的关键原则

6.1 当前机器上哪些分区能动，哪些绝对不要动

这台机器上：

绝对不要动的分区

整块 nvme0n1 都是 Windows 相关内容，重装 Arch 时不要格式化、不要改分区表：

  • /dev/nvme0n1p1：Windows EFI
  • /dev/nvme0n1p2：Windows MSR
  • /dev/nvme0n1p3：Windows 系统分区
  • /dev/nvme0n1p4：Windows Recovery
  • /dev/nvme0n1p5：Windows 数据分区

本次重装中只格式化这个分区

  • /dev/nvme1n1p2：当前 Arch 根分区，300G，ext4 → 目标改为 Btrfs

本次重装中保留不格式化的 Arch 分区

  • /dev/nvme1n1p1：Arch EFI，1G，当前挂载 /efi
  • /dev/nvme1n1p3：Arch /home，1.6T，当前 ext4

7.3 关于 swapfile 的处理

当前系统使用的是根分区上的 /swapfile，大小 24G。由于根分区会被重建，旧的 /swapfile 会消失。

因此新系统安装完成后，需要重新创建 swapfile，而不是尝试恢复旧文件。

如果新系统根分区仍为 Btrfs，建议按 Arch Wiki 的 Btrfs 方式重新创建 swapfile；不要简单照搬 ext4 上的旧做法。

  • 绝对不要动整块 nvme0n1 上的任何 Windows 分区
  • 本次重装只格式化 /dev/nvme1n1p2
  • /dev/nvme1n1p1 是当前 Arch 使用中的 EFI，继续挂载到 /efi，不要误挂到 /boot
  • /dev/nvme1n1p3 是单独的 /home 分区，除非你明确决定重建 home，否则不要格式化
  • fstab 必须以新系统生成的 UUID 为准
  • 旧 /swapfile 不要直接恢复，要在新系统上重新创建
  • 新系统用户 UID/GID 最好与旧系统一致
  • 当前没有看到 LUKS、LVM、Secure Boot 痕迹；如果后续确认启用了这些，再单独细化
  • 如果有 Docker、libvirt、PostgreSQL 等数据服务，恢复前务必先核对对应数据目录

15. 当前机器配置快照

以下信息来自当前系统现场，可作为重装时核对基准：

磁盘与分区

  • nvme0n1：476.9G，Windows 磁盘
    • p1：100M，vfat，UUID B8F3-B44C
    • p2：16M，MSR
    • p3：199G，NTFS，UUID 9AB0006BB0005065
    • p4：958M，NTFS，UUID 1CC0C7ABC0C78A0A
    • p5：276.9G，NTFS，标签“新加卷”，UUID D61471BB14719EE5
  • nvme1n1：1.9T，Arch 磁盘
    • p1：1G，vfat，UUID D300-6838，挂载 /efi
    • p2：300G，ext4，UUID 1f6b34bd-9b0e-4933-966c-6cbdfa1703ff，挂载 /
    • p3：1.6T，ext4，UUID f83c7b57-3c92-4555-b6f0-d12916801179，挂载 /home

启动项

  • Boot0003：GRUB，当前默认启动项
  • Boot0004：Windows Boot Manager
  • 当前 BootOrder：0003,0004,2001,2002,2003

当前 fstab 要点

  • 根分区挂载：/dev/nvme1n1p2 → /
  • EFI 挂载点：/dev/nvme1n1p1 → /efi
  • home 挂载：/dev/nvme1n1p3 → /home
  • 存在 /swapfile：24G

当前 GRUB 要点

  • GRUB_CMDLINE_LINUX_DEFAULT="loglevel=5 nowatchdog"
  • 当前未显式启用 GRUB_DISABLE_OS_PROBER=false

16. 下一步建议
